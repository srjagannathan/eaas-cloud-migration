Challenge 2 · Discovery

What Nobody Told Us

I had Claude roleplay each stakeholder — the SRE, the DBA, the batch job owner. In under five minutes, we had things that weeks of ticket-reading wouldn't have surfaced.

Hardcoded IP 192.168.10.45 — buried in the web app config, the batch DSN, and a mystery cron. All three workloads are coupled to that one IP. You can't move one without moving all three in the same window.
Shared NFS mount /mnt/nfs/reports/ — the invisible wire between the web app and the batch job. Our SRE only found it when the PDF downloads broke for an hour after a restart. It's now S3.
The warm-ping cron — curl http://192.168.10.45:5432 every five minutes. Nobody knew what it did. Turns out it was keeping pgBouncer alive. I killed it. ElastiCache handles sessions now.
Three years of credential sprawl — one shared Postgres admin password across five teams, living in spreadsheets and Confluence pages. That gets rotated at cutover. Every team gets their own read-replica endpoint.

The Calls We Made — and Why They Hold for 15 Years

ADR-001 · Migration Pattern
Lift-and-shift first. Phase 2 in writing.
Compliance residency closes in weeks, not quarters. Refactoring mid-migration introduces failure modes in a system we're still learning. We deferred the CTO's cloud-native goals by 90 days — not abandoned, committed. The four discovery blockers get resolved regardless of pattern. That's the deal we put in the memo.
ADR-002 · Revised on 15-year criteria
We changed three of our four service decisions after stress-testing for longevity
Our first pass optimised for cutover speed. Our second pass added ecosystem longevity, AI/ML readiness, and OLAP/OLTP separation as criteria. ECS Fargate, RDS, and a read replica for BI all failed that test. EKS, Aurora, and Redshift Serverless passed it.

How We Used Claude Code

Claude as a Force Multiplier — the Team Made the Calls

PreToolUse Hook
We enforce secrets policy at the tool level, not the prompt level
A CLAUDE.md prompt shapes Claude's preferences — probabilistic, not guaranteed. Our PreToolUse hook in .claude/settings.json blocks any literal credential in a .tf file unconditionally, before the write lands. Claude attempted to inline placeholder values twice during IaC generation; the hook caught both. We documented the distinction in ADR-003 — it's a design principle, not just a config choice.
Three-Level CLAUDE.md
Four people, one shared context — no verbal syncing required
One project-level file captures shared conventions — stack, secrets rule, commands. Three workload-level files (web-app/, batch/, reporting-db/) carry workload-specific constraints. Raymond, Venkat, Saurabh, and Srinath ran Claude sessions in parallel. The shared context meant Claude's outputs stayed consistent across all four streams without the team having to manually align.
Stakeholder Roleplay
We used Claude to stress-test assumptions, not replace judgment
Our team directed Claude to play the SRE, the DBA, and the batch job owner — each in character, each with their own agenda. The warm-ping cron, the NFS coupling, the shared credentials: none of that appears in a config file grep. It surfaced through the roleplay. Claude gave us the raw material; our architects decided what to act on and why. That distinction shows up in every ADR.
Plan Mode + Second-Pass Architecture
We ran two passes — Claude helped us pressure-test the first one
Our first architecture pass optimised for migration speed. We then used Claude to systematically challenge each service choice against a 15-year horizon — portability, AI/ML readiness, OLAP/OLTP separation. ECS Fargate, RDS, and the BI read replica all failed that second pass. The team made the reversal calls. Claude made the analysis faster and more rigorous.

Roadmap — Built Into the Architecture from Day One

Three Phases. Not One Big-Bang Rewrite.

Phase 2 · Within 18 Months
Kill the nightly batch. Move to real-time.
Nightly reconciliation is an on-prem constraint, not a sound architecture. Our Phase 2 target: Aurora CDC → Amazon MSK → stream processor on EKS. Intraday dashboards via Redshift Serverless. The batch job we built is already decomposed into functions — when we move to MSK, those become stream consumers, not a rewrite. We committed this to the CTO alongside the lift-and-shift decision. It's in writing.
Phase 3 · Year 2–3
AI layer in production. Global DB active.
Bedrock integration for fraud/anomaly detection and document intelligence — the hooks are in the architecture today. Aurora Global Database second region activated (us-west-2) for sub-second RPO across regions. pgvector queries running against transaction embeddings for risk scoring. OpenTelemetry feeding Grafana dashboards the SRE team actually trusts. None of this requires re-architecture — it's additive on the foundation we're shipping now.
Automate Discovery · Challenge 9
What we did manually should run in CI
Our team ran stakeholder roleplay manually to surface the four hidden dependencies. Challenge 9 (The Survey) does that automatically: parallel Task subagents, one per workload, explicit context per call, a coordinator that merges structured JSON output. The cross-workload coupling — the one finding that required knowing all three workloads simultaneously — is exactly what a coordinator pattern catches and a single-pass analysis misses. That belongs in CI, not in a team member's head.
